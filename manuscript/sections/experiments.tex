\noindent This section describes the trace-processing and evaluation protocol. The routes move from aggregate cluster usage to machine entities, cross-machine transfer, and an audited service cohort. All forecasting routes use chronological splits, with test windows held out from fitting and calibration.

\subsection{Dataset and preprocessing}

The experiments use Alibaba \texttt{cluster-trace-v2018}, which contains approximately 4000 machines over 8 days and provides machine-level utilization, container-level usage, and batch-workload tables \citep{alibabav2018trace}. The study derives an aggregate series from \texttt{machine\_usage}, a high-coverage machine cohort from the same archive, and a service cohort from \texttt{container\_meta} plus a full local mirror of \texttt{container\_usage.tar.gz}.

The preprocessing script reads machine-usage CSV records from the tar archive, drops invalid rows, aggregates by timestamp, and emits one-minute mean CPU utilization, mean memory utilization, and active-machine count. This keeps the aggregate route auditable, while the machine and service routes narrow the construct-validity gap. The service route uses a verified full mirror of \texttt{container\_usage.tar.gz}, removing the earlier low-\texttt{container\_id} prefix heuristic.

\subsection{High-coverage machine cohort}

The machine route tests whether aggregate conclusions survive a finer unit of analysis. It scans \texttt{machine\_usage.tar.gz}, selects the 12 machines with the highest row counts, aggregates each to one-minute means, and reindexes them onto the common global span. Missing minutes are marked and filled only after selection. The retained machines cover 93.7\%--94.7\% of the span, with a 5.8\% average imputation ratio.

This cohort is not a service-level benchmark; it is a controlled cross-entity check between the aggregate and service routes.

\subsection{Container service cohort}

The service route extracts application-level evidence from the official container tables. It uses \texttt{container\_meta.tar.gz} for container-to-\texttt{app\_du} membership and request sizes, then maps a full mirrored pass over \texttt{container\_usage.tar.gz} back to services. Service demand is aggregated to one-minute resolution using mean utilization and a CPU-core proxy defined by requested cores times observed CPU utilization.

This is therefore a much broader official-data extension, although it still should not be read as a full population census because the metadata prefilter intentionally excludes smaller and shorter-lived services (see Table~\ref{tab:service-cohort-stats}). The fixed rule requires at least 60 containers and 650{,}000 seconds of metadata span, then retains profiled \texttt{app\_du} groups with matched-container hit ratio at least 0.90 and observed-minute coverage at least 0.96. The final cohort contains 139 services, 19{,}419 selected containers, 19{,}269 matched containers, and a common 11{,}520-minute span; the top-10 overlap audit retains all earlier targeted services.

\subsection{Forecasting and policy configuration}

The forecasting benchmark implements persistence, moving average, linear regression, random forest, and a feed-forward MLP regressor. It also includes \texttt{UA-MSTCN-Lite}, a trainable uncertainty-aware surrogate using multi-scale history features and a quantile-aware tree ensemble. These models validate the harness and define performance floors for later deep temporal variants.

The revision adds quantile linear regression (QLR) as a simpler uncertainty-aware baseline. QLR emits $P50$ and raw $P90$ forecasts for 1-, 5-, and 10-minute horizons; artifacts retain raw $P90$, raw crossing counts/rates, and guarded $\max(P90_{\mathrm{raw}},P50)$ outputs. QLR fitting uses only the train-plus-validation prefix, and the test split is used only for evaluation. After pilot, calibration, and guarded preflight, QLR was also run on the full 139-service cohort as conservative boundary evidence rather than as a new superiority claim.

The aggregate series uses a chronological $70/15/15$ split. The first 85\% supports training and validation; the final 15\% is held out for test evaluation. The default context window is 60 minutes, horizons are 1, 5, and 10 minutes, and forecasting quality is measured with MAE, RMSE, MAPE, and pinball loss.

The autoscaling simulator compares a reactive threshold policy, a lagged target-tracking policy, and a predictive policy using median and high-quantile forecasts with cooldown and step-size constraints. All policies act in the same required-capacity space; the predictive route uses the calibrated upper quantile plus the configured scale-out margin.

\subsection{Extended analysis protocol}

Beyond the core tables, the protocol tests context sensitivity (30, 60, and 120 minutes), 5-minute regime behavior, empirical quantile calibration, policy ablations, cooldown/margin sensitivity, 3-fold cross-machine zero-shot transfer, and service-level forecasting plus control. For service policy, the resampling unit is \texttt{service}. The primary statistic is the equal-weighted mean predictive-minus-reactive delta; a load-weighted mean using the service CPU-cores proxy is auxiliary. Stability is assessed with 10{,}000 paired bootstrap resamples over services and seed 20260311: intervals fully below zero indicate improvement, intervals fully above zero indicate degradation, and intervals crossing zero are not stable. The same pre-fixed rule governs whether the broad audited cohort replaces earlier targeted evidence.

The broad service route also computes mean load, coefficient of variation, 95th-percentile-to-mean burstiness, lag-1 autocorrelation, imputation ratio, and usable observed-minute count. Rank-based median splits report policy deltas, and a compact taxonomy groups services into SLA-improved/over-improved, SLA-improved/actions-worsened, over-improved/SLA-worsened, all-worsened, near-neutral, and other mixed cases. Near-neutral requires all three absolute policy deltas to fall below the cohort-specific first-quartile absolute-delta threshold. The machine cohort reuses the same leakage-free setup for direct comparison.

\subsection{Artifacts and environment}

The pipeline writes outputs to \texttt{paper-workbench/experiments/results/}. The metrics contract is fixed by \texttt{metrics.csv}, policy series by \texttt{policy\_series.csv}, and figures are rendered from artifacts rather than edited manually.

The experiments used TeX Live 2023, \texttt{pdflatex}, \texttt{latexmk}, and a Python environment with \texttt{pandas}, \texttt{scikit-learn}, \texttt{pyyaml}, and \texttt{matplotlib}.
